ARC-FMS · Concept Walkthrough

Fee Management, Reimagined

Redesigning the institute experience by making critical workflows faster, more intuitive, and easier to navigate. The focus is on surfacing key information — collections, transactions, reports and student records — within the dashboard, reducing dependency on manual reports and repetitive tasks.
What it containsSeven sections — GILE navigation, stats & collections, institute setup, transactions, reports and an AI assistant. Each shows the live screen (before), the problem, and a clickable prototype (after) — ending with one verified production data bug.
Live today — a multi-campus institute on the current ARC dashboard

1 · GILE Navigation

Group › Institute › Location › Education. A large chain can span dozens of campuses × 3 boards — this is where navigation hurts most.

The problem: selecting a GILE happens in a pop-up “Search & Filter” modal — ~5 clicks, it covers the screen, and the choice resets when you move between Statistics / Transactions / Reports. Worse: once inside an institute you can’t tell which GILE you’re viewing.

2.2 Defaulter Worklist Enhancement

The problem: the pending total is a number on the dashboard; the who-owes list is a downloadable report; reminders are a separate scheduled broadcast in Communications — the pieces exist but aren't in one place, so you can't see a live list and remind one student (or one overdue bucket) on the spot. Fix: one live, actionable screen — each defaulter (amount · class · fee header · days overdue) with Send reminder per student, bulk multi-select, filter by overdue bucket (0–30 / 31–60 / 60d+), or “remind all overdue”. No report download, no separate scheduler.

3 · Institute Setup

Kill the yearly rebuild of classes / fee headers / financing modes / payment pages.

3.1 Clone Previous Year Setup Full flow

What we have today: every new academic year the institute rebuilds all config from scratch — classes, fee headers, financing modes, payment pages — even though last year's is almost identical.

The gap: there's no "carry forward" — weeks of manual re-entry each July, with a real chance of a fee or a page being missed or mistyped.

How this helps: a "Clone Setup" action copies last year's classes/pages into the new AY (fresh URLs, reset expiry, amounts flagged to review) — config only, never student data.

6 · AI Assistant

6.1 Dashboard + Floating Assistant AI

The problem: the dashboard shows numbers, but acting on them means navigating to the right screen and setting it up yourself — insight and action sit apart. Fix: a grounded, scoped assistant that answers fixed queries inline (numbers from data, never invented) or opens the real screen prefilled — ending in an action.

6.2 Financial Assistance Recommendation AI

What we have today: a parent who stops paying is just a row in the pending report — a reliable, long-time payer looks identical to a chronic defaulter, and both get the same generic reminder.

The gap: nothing separates “won't pay” from “can't pay right now” — the parent who paid on time for years and just bounced this term slips into a hard default, or a dropout, when a plan would have kept them.

How this helps: it flags exactly the reliable-but-now-slipping parents (strong on-time history + current arrears, extra weight on a bounce) and lets the institute privately offer an editable EMI / split plan in one click — the decision, and the offer, stay with the institute.

7 · Communications

Reach parents at scale — the right segment, on the right channel, without dropping anyone.

7.1 Broadcast Center Concept

The problem: notices & reminders go out as a scheduled broadcast to “All students” from a separate screen — you can't finely segment the audience, can't see who's actually reachable, and parents with no phone/email quietly fall through. Fix: a Broadcast Center (from the Master dashboard / Communications) — pick a segment (All · Pending Fee · EMI · PG · Auto-Debit, with live counts across ~2,500 parents) + class, send on WhatsApp / SMS / Email with smart fallback, compose with merge tokens & an attachment, and a reachability panel that surfaces every unreachable parent — including Student-ID-only ones — with a page-notice + QR fallback so no one is dropped.

7.2 Rapid Payment Pages — self-serve page creation Concept

The problem: creating a GrayQuest payment page today means hand-writing INSERTs across ~9 config tables in DBeaver — generating UUIDs, crafting each SDK / fee-group JSON block by hand, one page at a time. Slow, error-prone, engineer-only. Fix: a plain-English, split-screen studio where the Configuration team builds pages by clicking — the live payment page (exactly as a parent sees it) renders on the left as the wizard drives it on the right, and every choice quietly maps to the right table underneath.

  • Three page types: FMS (per class, batch across Class 1–12), ERP (institute-wide, webhook + downloadable integration credentials), and Master link (a cascading dropdown that routes each student to the right page).
  • Fees fixed or parent-entered; payment options (EMI · Card & UPI · Auto-Debit) built by ticking fee headers — the fee-group expression is generated silently; form, look, expiry & fee receipt (single or per-class).
  • Multiple pages per class (e.g. an EMI-only + a PG-only page) sharing one credential, with every slug editable — and a one-click “Create Master link for these links”.
  • Master link from Excel: upload a sheet (columns → link) and it builds the dependent-dropdown JSON — byte-identical to the team's existing Python script.

8 · Verified Data Bug

Not a mockup — confirmed against the production database.

Fee Header Report — phantom pending = late fine Bug

The “Fee Header Due Amount” sheet computes Paid Amount from the base fee (excludes the collected late fine), while Total Payable includes the fine — so every student who paid with a fine shows a false Pending = the fine.

  • Proof: BHAVADHINI S (2535S0003) — “Fee Header Paid” sheet shows ₹62,750 paid; “Due Amount” sheet shows ₹62,500 paid / ₹250 pending. Source DB (paid_amount) confirms the fine was collected.
  • Impact: 19,811 orders · ₹1,18,02,140 (~₹1.18 Cr) of already-collected late fees falsely reported as pending, system-wide.
  • Fix: the Due-Amount sheet's Paid Amount must use the actual collected amount (as the Paid sheet already does).
Dashboard — CORRECT: ₹62,750 · SETTLED
Report “Due Amount” sheet — WRONG: ₹250 pending
Same student 2535S0003 (ref GQDT-J1HMEERPQMB): the dashboard shows the full ₹62,750 paid · SETTLED — correct, it includes the ₹250 fine. But the report's Due-Amount sheet (rows 4 & 5) counts Paid as 62,500 and shows Pending 250 — the phantom. Only the report is wrong; the dashboard reads the real collected amount.
